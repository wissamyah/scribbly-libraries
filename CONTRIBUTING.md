# Contributing to Scribbly Libraries

Thanks for sharing a library. This document is the contract between submitters and the registry. Every PR is checked against the **automated rules** below (CI rejects before a maintainer ever sees it) and the **content guidelines** (subjective — enforced by reviewers).

## PR walkthrough

1. **Export your library from Scribbly.** Open the Library sidebar → ⋯ → *Export*. You'll get a `library.scribblylib` file.
2. **Fork this repo.** Locally:
   ```bash
   git clone git@github.com:<your-handle>/scribbly-libraries.git
   cd scribbly-libraries
   git checkout -b add-<library-slug>
   ```
3. **Create a submission directory** at `submissions/<your-github-handle>/<library-slug>/`. The handle directory **must match your GitHub login** (lowercase). The slug is your library's identifier — kebab-case, English, short.
4. **Add two files** to that directory:
   - `library.scribblylib` (the file you exported)
   - `meta.yaml` (see [the template below](#metayaml-template))
5. **Do not commit `preview.png`** — it is generated by CI on merge.
6. **Open a PR.** CI runs `scripts/validate.ts`. Fix anything it complains about. Once green, a maintainer reviews against the content guidelines.

## Automated rules (CI rejects before human review)

| # | Rule | Enforced by |
|---|---|---|
| 1 | The `.scribblylib` file parses as valid JSON | `JSON.parse` |
| 2 | Matches the `.scribblylib` v1 JSON Schema (shape, enums, required fields, per-type required field branches) | `ajv` against `schemas/scribblylib.schema.json` |
| 3 | `meta.yaml` matches the metadata JSON Schema | `ajv` against `schemas/meta.schema.json` |
| 4 | `library.scribblylib` ≤ **512 KB** | `fs.statSync` |
| 5 | Each `libraryItems[].elements` is **self-contained** — every `groupId`, `frameId`, `containerId`, and arrow `binding.elementId` refers to another element **in the same item** | `validate.ts` |
| 6 | The library has **≥ 3 items** | length check (also enforced by schema) |
| 7 | Each item is either ≥ 2 elements **or** a single non-trivial element (text with non-empty content, freedraw with > 10 points, etc.) — blocks "library of single arrows" submissions | `validate.ts` |
| 8 | All `seed` values are integers | type check (also enforced by schema) |
| 9 | All text content (in-item text labels, library name, description) passes a basic-English heuristic: ≥ 90 % of characters in basic Latin + common punctuation | regex in `validate.ts` |
| 10 | `meta.license` is one of the allowlisted SPDX ids: `MIT`, `Apache-2.0`, `CC0-1.0`, `CC-BY-4.0`, `CC-BY-SA-4.0` | enum in schema |
| 11 | `meta.name` ≤ 60 chars; `description` ≤ 280 chars; ≤ 8 tags from the controlled vocabulary | length / enum checks in schema |
| 12 | Slug `<github-handle>/<library-slug>` is unique against existing `submissions/` and against git history (blocks handle squatting after a delete) | `validate.ts` + filesystem + `git log` |
| 13 | No `image` elements (base64 image elements are not allowed in marketplace submissions) | schema (`type: image` is rejected) |
| 14 | The submission directory contains **only** `library.scribblylib` and `meta.yaml` (no stray files, no `preview.png` commits) | `validate.ts` |
| 15 | The author's GitHub handle (PR author) **matches the directory name** under `submissions/` | `validate.ts` (when `GITHUB_ACTOR` is available) |
| 16 | When updating an existing slug, `meta.version` strictly increases (strict semver) | `validate.ts` against the previously published manifest |
| 17 | Re-publishing the same `meta.version` with different bytes is rejected (SHA-256 stability) | `validate.ts` against the previously published manifest |

If CI is red, **read the log first**. The validator prints the offending file and field for every failure.

## Content guidelines (subjective — reviewer enforces)

These are borrowed from Excalidraw's published library guidelines, tightened slightly. The CI validator can't enforce them — a human will, and approval is at the maintainer's discretion.

1. **English only** in titles, descriptions, tags, and any in-item text labels.
2. **Original work**, or a clearly attributed remix with permission compatible with your declared license. No copies of other published libraries without significant added value.
3. **Broad utility** — the library is useful to someone who is not the author. Pure personal-use libraries (your team's org chart, your D&D campaign map) will be rejected.
4. **Thematic coherence** — items in one library belong together. Mixed-bag libraries are split or rejected.
5. **Grouped items** — multi-element items must be saved as groups so they insert as a single unit. (The validator can't always tell intent; reviewers enforce this.)
6. **No trivially reproducible content** — single arrows, single rectangles, single text labels. The validator's "non-trivial" check catches the obvious cases; the reviewer catches the rest.
7. **Safe content** — no porn, gore, hate symbols, real-person likenesses without consent, brand logos used in misleading ways, or content designed to deceive (fake UI mocking real bank login pages, etc.). Removal is unilateral.

Approval is not guaranteed. If your PR is closed without merge, the reason will be in the PR comments.

## `meta.yaml` template

```yaml
# Human-readable library name. English only. Up to 60 chars.
name: Server Rack

# 20–280 chars. What this library contains and who it's for.
description: Datacenter rack units, switches, and cables for infrastructure diagrams.

# Strict semver. Bump this on every published update to this slug.
version: 1.0.0

author:
  # Your GitHub handle — lowercase. Must equal the directory name under submissions/.
  handle: wissam

  # Optional. Display name shown on the gallery card.
  displayName: Wissam Y.

# Optional. Link to the library's homepage or source.
homepage: https://example.com/server-rack

# SPDX id from the allowlist.
license: CC-BY-4.0

# 1–8 tags from the controlled vocabulary in schemas/meta.schema.json.
tags:
  - infrastructure
  - diagrams

# Optional. id of the libraryItem to render as the gallery thumbnail.
# Defaults to the first item.
# previewItemId: 00000000-0000-0000-0000-000000000000

# Optional. Set to true to deprecate a previously published version.
# deprecated: false
# deprecationNote: superseded by foo/server-rack-v2
```

## Tag vocabulary

The current allowlist is in `schemas/meta.schema.json` under `properties.tags.items.enum`. Adding a new tag requires a separate PR that updates the schema and this file.

## Updating an existing library

To publish a new version of a library you already own:

1. Edit `submissions/<your-handle>/<library-slug>/library.scribblylib` (drop in a freshly exported file).
2. **Bump `meta.version`** to a strictly greater semver. CI rejects same-version-different-bytes and version regressions.
3. Open a PR. Reviewer reapplies the content guidelines.

## Removal

To remove your own library: open a PR that deletes the directory. To report someone else's: use the in-app **Report** button (it pre-fills a GitHub issue) or open one directly. Maintainers may unilaterally revert merged submissions that violate the content guidelines.
