# Scribbly Libraries

Community-curated `.scribblylib` packs for [Scribbly](https://scribbly.app) — the collaborative hand-drawn whiteboard app.

This repo is a **PR-based registry**. Submissions live under `submissions/<github-handle>/<library-slug>/` and are approved by maintainer review. On merge, a GitHub Action regenerates `libraries.json` and publishes it (together with the `.scribblylib` files and rendered previews) to <https://libraries.scribbly.app/>.

Scribbly's in-app **Browse** tab reads `libraries.json` directly from that URL. There is no backend, no database, no upload form — the PR queue is the moderation surface.

## How to submit a library

1. Read [CONTRIBUTING.md](./CONTRIBUTING.md) first. It explains both the automated rules (CI rejects before review) and the content guidelines (subjective, enforced by reviewers).
2. Export a `.scribblylib` from Scribbly (Library sidebar → ⋯ → Export).
3. Fork this repo and add **two files** under `submissions/<your-github-handle>/<library-slug>/`:
   - `library.scribblylib` — the file you exported
   - `meta.yaml` — human-edited metadata (see [`schemas/meta.schema.json`](./schemas/meta.schema.json))
4. Open a PR. CI will validate. Fix anything CI complains about.
5. A maintainer reviews against the content guidelines. Once merged, your library appears in the gallery on the next publish run.

Non-developers submit via Scribbly's in-app **Submit to gallery** button, which opens a structured [Issue Form](.github/ISSUE_TEMPLATE/submit-library.yml). When the issue is created, a GitHub Action ([`intake.yml`](.github/workflows/intake.yml)) auto-scaffolds the folder, opens a PR, and comments back with the PR link. The submitter waits, the maintainer reviews + merges.

## Reviewing submissions (maintainer flow)

Day-to-day, you should only ever need to do one thing:

1. A submission issue lands → the bot opens a PR → CI runs `validate.yml`.
2. Open the PR on github.com. Skim the diff: `meta.yaml` for sanity, `library.scribblylib` exists.
3. (Optional) Verify by hand: `gh pr checkout <n>` then `npm run validate -- submissions/<handle>/<slug>` locally. Or import the file into Scribbly to preview visually.
4. **Merge.** `publish.yml` regenerates `libraries.json` and ships it.
5. Close the source issue (the PR's `Closes #N` auto-closes it on merge).

If the submitter made a mistake (bad slug, missing file, attachment isn't valid JSON), the bot comments on the issue with what's wrong and doesn't open a PR. The submitter edits the issue or comments `/retry`, and the bot tries again.

### Legacy / fallback: `npm run review`

For older free-form issues (pre-Issue-Form), or when the bot can't help, scaffold by hand:

```bash
npm run review -- <issue-number>            # scaffold + validate locally
npm run review -- <issue-number> --auto-pr  # also create branch + open PR
npm run review -- <issue-number> --force    # overwrite an existing slug dir
```

This is the manual escape hatch — under the bot-driven flow you shouldn't reach for it.

## Removal

To remove a library you submitted, open a PR that deletes the `submissions/<handle>/<slug>/` directory. To report someone else's library, use the **Report** button in the in-app gallery (it pre-fills a GitHub issue) or open an issue directly. Maintainers may unilaterally revert merged submissions if they violate the content guidelines.

## Publishing pipeline

On every push to `main`, `.github/workflows/publish.yml`:

1. Re-runs typecheck + unit tests + `validate.ts` against every submission (defense in depth — the PR already validated, but this catches anything that bypassed CI).
2. Runs `npm run build-manifest`, which walks `submissions/`, renders a preview PNG for each library via `@napi-rs/canvas` + `@scribbly/renderer`, copies the `.scribblylib` to a versioned path, computes its SHA-256, and writes `dist/libraries.json`.
3. Uploads `dist/` as a Pages artifact and deploys to <https://libraries.scribbly.app/>.

`publishedAt` is preserved across rebuilds; `updatedAt` advances on every run.

### One-time Pages setup

In repo Settings → Pages, set **Source: GitHub Actions**. No custom branch needed. The first push to `main` will provision the Pages site.

To use the production hostname `libraries.scribbly.app`, add a CNAME record pointing at the GitHub Pages domain and configure the custom domain in Pages settings. Until then the site is reachable at the default `<owner>.github.io/<repo>/` URL — set Scribbly's `VITE_LIBRARY_REGISTRY_URL` to match during the transition.

## Versioning a library

Each library has a `version` in its `meta.yaml`, strict semver. Bumping it is the **only** way to publish an update under the same slug. Old versions stay reachable at their pinned download URL so deep links don't break. Re-publishing the same version with different bytes is rejected by CI.
